## 2. 当前已完成的前端与后端修复说明

### 2.1 登录界面

登录界面已经调整为适合 10 人以下小团队使用的账号入口：

- 登录页只显示用户名/邮箱与密码，不再显示 agent。
- 最近账号只保存邮箱，不保存密码。
- 切换最近账号时会清空密码，必须由用户重新输入。
- 登录页展示后端与 OpenClaw 的连接状态，便于判断 `/api` 代理或 Gateway 是否可用。
- 新账号注册后才会进入首次 agent 配置流程。

这个方向是正确的：对小团队来说，入口应该像“团队工作台登录”，而不是“agent 选择器”。agent 属于登录后的个人资源，不应该在登录前暴露。

### 2.2 首次初始化流程

当前逻辑已经改为：

- 注册成功后写入一次性的 `xuanzhi.agentSetup.pending.<userId>` 标记。
- 只有该用户首次进入、且 agent 尚未有 profile 时，才自动弹出配置界面。
- 普通登录不会因为 profile 为空就反复弹配置。
- 配置保存后清除 pending 标记。

这修复了“每次登录都像初始化”的体验问题。后续更稳的方式是把 `profileCompletedAt` 放到后端用户或 agent 记录中，而不是依赖浏览器 localStorage。

### 2.3 用户与 agent 隔离

当前后端已经按用户隔离 agent：

- `/api/agents` 只返回当前登录用户自己的 agent。
- 登录或获取 `/api/auth/me` 时，会确保当前用户拥有自己的本地 agent。
- 后端运行 OpenClaw 会使用当前 agent 的 `userId`、`profile`、`workspace`。
- 切换用户后不会把 A 用户的 agent 显示给 B 用户。

这对小团队很关键。它让前端呈现“我的玄知 Agent”，而不是一个共享 agent 池。

### 2.4 OpenClaw workspace 位置

当前 workspace 已经改为与 OpenClaw 自身 workspace 并列：

```text
/home/lin123/.openclaw/
  workspace/
  workspace-xuanzhi-user_xxx/
  workspace-xuanzhi-user_yyy/
```

这比把 workspace 放到项目目录或错误的子目录中更合理，因为 OpenClaw 自身可以直接识别和使用这些目录。

### 2.5 用户 profile 同步给 OpenClaw

此前的问题是：用户注册后的姓名、偏好、角色等信息只存在玄知后端内存里，OpenClaw agent 本身不知道这些配置。因此问 OpenClaw 时，它无法知道用户填写过什么。

当前修复方式：

- 创建或使用 OpenClaw agent 前，会尝试同步用户 profile。
- 同步文件为 `USER.md` 与 `AGENTS.md`。
- OpenClaw 实测不支持任意 `xuanzhi-profile.json` 文件名，因此已改为只同步 Markdown 文件。
- 如果 profile 文件同步失败，聊天不会直接失败，而是记录 `agent.profile.sync_skipped` 事件并继续对话。

### 2.6 首条消息中的任务前缀

此前第一句话被后端拼接为：

```text
[任务ID: task_xxx]
[任务意图: meeting]
用户原始问题
```

这会污染用户真实问题，也会让 OpenClaw 误以为这些是用户要表达的内容。当前已经改为直接发送用户原文，不再拼接任务 ID 与任务意图。

## 3. 稳定性与可用性评估

### 3.1 当前最大稳定性风险：内存存储

后端仍然使用 `MemoryStore`。这意味着：

- API 重启后，用户、token、任务、消息、agent 映射都会丢失。
- OpenClaw 里的真实 agent 仍存在，但玄知后端可能失去对应关系。
- 用户重新注册或登录后，可能产生新的本地 agent 映射。
- 10 人以下小团队也会遇到“服务重启后数据消失”的严重问题。

建议优先级：最高。

建议方案：

- 短期：SQLite + migration，保存 users、sessions、agents、tasks、messages、events。
- 中期：增加 OpenClaw agent reconciliation，按稳定的 `userId/workspace` 找回已有 agent。
- 长期：如果要多人长期使用，再切到 Postgres。

### 3.2 SSE token 放在 URL 查询参数中

当前前端使用原生 `EventSource`，因为不能设置 Authorization header，所以 stream URL 中带 `?token=`。

风险：

- token 可能出现在代理日志、浏览器历史、错误日志中。
- 小团队内部使用可以暂时接受，但不能作为长期生产方案。

建议：

- 使用短时效 stream token，只能访问一个 task stream。
- 或使用 `fetch-event-source` 一类方案，通过 Authorization header 建立流。
- 后端保留 task owner 校验，这是已经做对的部分。

### 3.3 Profile 完成状态应由后端持久化

当前首次配置是否弹出由前端 localStorage 标记控制。它解决了当前体验，但仍有边界问题：

- 换浏览器或换电脑后，pending 标记不存在。
- localStorage 被清空后，状态不可靠。
- 后端无法明确知道 profile 是否完整。

建议在 agent 或 user 上增加：

- `profileCompletedAt`
- `profileVersion`
- `profileSyncedAt`
- `profileSyncStatus`

前端只读取后端状态决定是否弹出配置。

### 3.4 OpenClaw 连接状态需要进入主界面

登录页展示 Gateway 状态是好的，但用户进入应用后也需要看到：

- 当前 agent 是否 connected / idle / running / error。
- 当前 workspace 路径。
- 最近一次 profile 同步状态。
- 最近一次 OpenClaw 错误。

建议在主界面顶部或 agent 设置中增加一个低干扰的状态区，不要做成大面积告警。默认只显示一个状态 pill，异常时展开详情。

### 3.5 空响应与超时处理还需要更严格

OpenClaw 流式响应有 120 秒超时。当前如果没有拿到最终文本，存在“任务完成但回答为空”的风险。

建议：

- 如果超时且没有任何 assistant 文本，应标记 task/message 为 failed 或 degraded。
- 前端展示“OpenClaw 未返回内容，请重试或检查 Gateway”。
- 给用户一个重试按钮，重试时复用同一个 task 上下文。

### 3.6 前端构建包体积偏大

当前 Web 构建通过，但 Vite 提示主 chunk 约 827 KB，超过 500 KB 建议值。

建议：

- 对 chat、markdown 渲染、文件/工件预览、设置页做路由级懒加载。
- 配置 `manualChunks`，把 React、UI 组件、Markdown/高亮依赖拆分。
- 延迟加载不在首屏需要的编辑器、预览器和大型图标集。

## 4. 前端进一步优化建议

### 4.1 登录页设计方向

适合 10 人以下团队的登录页应该是安静、可靠、清晰的工具入口：

- 左侧或顶部保留产品识别与连接状态。
- 右侧是账号登录表单。
- 最近账号列表只显示邮箱、头像首字母、最后登录时间。
- 不显示 agent、不显示 workspace、不显示内部调试字段。
- Gateway 异常时给出明确动作：启动 API、检查 OpenClaw、重新检测。

建议增加：

- “忘记密码/联系管理员”入口。
- 最近账号删除按钮。
- 键盘可访问的账号切换。
- `aria-live` 朗读 Gateway 状态变化。

### 4.2 首次配置页

当前首次配置承担了“创建自己的 agent”的语义，建议进一步明确成 3 步：

1. 个人信息：姓名、称呼、角色、团队。
2. Agent 偏好：语气、工作方式、常用任务类型、语言。
3. OpenClaw 配置确认：workspace 路径、模型、同步状态。

保存后前端应展示一句明确反馈：

```text
你的 Agent 已创建，并已同步到 OpenClaw。
```

如果同步失败，则展示：

```text
Agent 已创建，但 profile 暂未同步到 OpenClaw。你仍可以开始对话，稍后可在设置中重试同步。
```

### 4.3 主工作台

主界面建议围绕“小团队每天真的会用”的信息组织：

- 当前用户的 Agent 状态。
- 最近任务。
- 当前对话。
- 文件/工作区。
- Agent 设置。

不要把页面做成营销型 dashboard。这个产品的核心是“我登录后直接进入自己的 Agent 工作台，并能稳定使用它完成任务”。

建议增加：

- 顶部状态 pill：API、OpenClaw、Agent、Profile sync。
- 任务列表筛选：运行中、失败、最近完成。
- 失败任务的重试入口。
- Agent 设置中的“重新同步 profile 到 OpenClaw”按钮。
- Workspace 路径复制按钮。

### 4.4 Agent 设置

当前 profile 修改能力还不应只停留在名称/角色。建议复用首次配置表单，让用户之后可以完整修改：

- 名称/称呼
- 角色
- 工作偏好
- 常用任务类型
- 默认语言
- 模型
- workspace

设置页保存后应立即触发 OpenClaw profile 同步，并展示同步结果。

### 4.5 错误反馈

需要把后端事件变成前端可理解的用户反馈：

| 后端事件 | 前端表现 |
| --- | --- |
| `agent.profile.synced` | 设置页显示同步成功 |
| `agent.profile.sync_skipped` | 非阻塞黄色提示，可重试 |
| Gateway offline | 顶部状态异常，任务输入框旁提示 |
| task failed | 消息区域显示失败卡片与重试按钮 |
| assistant timeout | 显示超时说明，不应显示为空回答 |

### 4.6 响应式与可访问性

建议补充：

- 登录页、配置页、主工作台的移动端截图回归。
- 表单错误提示与输入框 `aria-describedby` 关联。
- 登录、注册、配置弹窗的 focus management。
- 所有图标按钮有 tooltip 或 aria-label。
- 连接状态变化使用 `aria-live="polite"`。

## 5. 其余组件建议

### 5.1 后端数据层

优先补齐持久化。建议第一阶段不要过度设计，SQLite 已足够支持 10 人以下小团队：

- users
- sessions
- agents
- tasks
- messages
- events
- artifacts

同时增加一个启动时 reconciliation：

- 扫描本地 agents。
- 读取 OpenClaw agent 列表。
- 按 `gatewayAgentId`、`workspace`、`userId` 找回映射。
- 对孤儿映射给出管理端提示，不自动删除。

### 5.2 OpenClaw 集成

建议把 OpenClaw 能力封装为明确的 service contract：

- health check
- list agents
- create/find user agent
- sync profile files
- send message
- stream response
- delete/reconcile agent

并为文件同步加 contract test，确保只使用 OpenClaw 支持的文件名。当前实测 `USER.md` 与 `AGENTS.md` 可用，任意 JSON 文件名不支持。

### 5.3 任务与消息

建议区分：

- 用户原始输入
- 系统内部 metadata
- task intent
- OpenClaw 上下文注入

不要再把任务 ID、意图等内部字段拼接进用户文本。需要给 OpenClaw 的上下文应通过 system/profile 文件或明确的 metadata 通道传递。

### 5.4 开发体验

建议增加统一启动与检查脚本：

```powershell
pnpm dev
pnpm health
pnpm test:all
```

`pnpm health` 建议检查：

- API port 是否监听。
- Web port 是否监听。
- OpenClaw Gateway 是否 healthy。
- 当前 `.env` 是否缺关键配置。
- workspace root 是否存在。

### 5.5 测试体系

建议继续补充：

- Playwright E2E：注册、首次配置、首条消息、退出、另一个用户登录。
- 跨用户隔离 E2E：B 用户不能看到 A 用户 agent/task/message。
- API restart 测试：持久化完成后验证重启不丢用户与 agent 映射。
- OpenClaw contract 测试：profile 文件同步、workspace 路径、stream 事件。
- 失败恢复测试：Gateway offline、profile sync failed、assistant timeout。

## 6. 优先级路线图

### P0：必须优先完成

- 引入持久化存储，替换纯 `MemoryStore`。
- 后端持久化 profile 完成状态。
- 空响应/超时任务标记为失败或可重试。
- 主界面展示 OpenClaw/Agent/Profile sync 状态。

### P1：强烈建议

- SSE 改为短时效 stream token 或 Authorization header 流式方案。
- Agent 设置页支持完整 profile 修改与重新同步。
- 增加 Playwright 端到端测试。
- 添加统一 dev/health 脚本。

### P2：体验优化

- 登录页最近账号支持删除与最后登录时间。
- Web 构建拆包，降低首屏 JS 体积。
- 移动端与可访问性回归。
- 管理员视图、技能市场、Agent 技能管理可以作为后续功能规划推进。

## 7. 总体判断

当前项目已经从“登录后容易串 agent、初始化信息不同步、首条消息被污染、workspace 位置错误”的状态，修到一个可以连接 OpenClaw 并完成基础使用的状态。

下一步真正影响可用性的不是界面装饰，而是三个基础能力：

1. 数据持久化。
2. OpenClaw 状态可见与可恢复。
3. 用户 profile 的后端状态化与可重新同步。

把这三件事补上后，这个系统会更接近一个适合小团队长期使用的前端工作台，而不是只能在单次开发会话里跑通的 demo。
