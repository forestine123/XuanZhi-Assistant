# 玄知助手第一版开发计划表

**目标周期：** 7 天  
**团队规模：** 4 人  
**版本目标：** 做出能本地启动、能登录、能创建任务、能执行 Agent/mock/OpenClaw、能展示过程、能审批、能落库的第一版。  
**质量要求：** 不追求完整平台化和高完成度 UI，优先保证主流程能跑通、数据能持久化、演示稳定。

---

## 1. 第一版范围

### 必须完成

- 用户可以登录、注册、退出。
- 用户可以创建任务并发送消息。
- 后端使用数据库持久化用户、任务、消息、事件、产物、审批、Agent。
- Agent 执行可以走 OpenClaw Gateway；Gateway 不可用时可以回退 mock。
- OpenClaw 插件可以向后端写入 event、artifact、approval、task status。
- 前端可以展示任务列表、聊天消息、Timeline、Artifact、Approval。
- 用户可以确认或拒绝审批。
- 后端重启后，历史任务和任务详情仍可查询。
- 本地可以通过 README 中的命令启动 API、Web、OpenClaw Gateway。

### 暂不完成

- PostgreSQL 生产部署。
- Redis、队列、对象存储。
- 完整多租户、团队空间、复杂 RBAC。
- 审计日志和凭证管理。
- 真实日历创建、真实邮件发送、真实业务系统修改。
- 知识库向量检索、文件解析。
- 代码写入、测试执行、PR 创建。
- 多 Agent 调度和复杂插件市场。

## 3. 统一技术决策

| 决策项 | 第一版选择 | 原因 |
|---|---|---|
| 数据库 | SQLite | 一周内最快落地，避免 Docker/PostgreSQL 环境成本 |
| ORM | 暂不引入重型 ORM | 先用轻量 SQL 封装，减少迁移成本 |
| Agent Runtime | OpenClaw 优先，mock fallback | 保证演示不被 Gateway 阻塞 |
| 实时通信 | SSE | 项目已有基础，适合任务过程单向推送 |
| 权限模型 | userId 隔离 + admin role | 先满足多用户基本隔离 |
| 插件鉴权 | 服务 token | 插件不传 userId，后端按 taskId 反查归属 |

## 6. 联调主流程

第一版只验收下面这一条主流程：

1. 启动 API。
2. 启动 Web。
3. 启动 OpenClaw Gateway；如果 Gateway 不稳定，允许 fallback mock。
4. 用户 A 登录。
5. 用户 A 输入：`下周三上午帮我约张三开项目复盘会`。
6. 后端创建 task 和 user message。
7. Agent 或 mock 生成事件、计划 Artifact、会议草稿 Artifact、审批请求。
8. 前端展示聊天、Timeline、Artifact、Approval。
9. 用户 A 点击确认。
10. 后端更新 approval 和 task 状态。
11. 前端显示任务 completed。
12. 重启 API。
13. 用户 A 再次登录后仍能看到历史任务。
14. 用户 B 登录后看不到用户 A 的任务。

- 原计划要求在任务详情页内联展示 `AgentTimeline`、`ArtifactPanel`、`ArtifactViewer`；当前前端 contract 测试明确要求这些组件不存在，说明当前产品方向已经从“聊天里塞过程和产物”转向“更干净的聊天界面 + 审批靠近输入区”。
- 原计划要求 seed `admin@example.com`、`user-a@example.com`、`user-b@example.com`；当前需求已经要求删除 A/B 测试 agent，并以真实注册用户为主。后续如保留 seed，也应只用于开发环境，并且不能自动创建真实 OpenClaw agent。
- 原计划要求 message 中包含 `taskId`、`intent`、`userInput`；当前已经确认不能把 `[任务ID]`、`[任务意图]` 拼进用户消息。内部 metadata 应通过独立字段、系统上下文或 profile 文件传递。
- 原计划强调 Gateway 不可用时 mock fallback；当前用户重点是验证真实 OpenClaw 连接。如果恢复 fallback，也要在 UI 上明确标注“当前是 mock”，避免误以为已连接真实 OpenClaw。

### 13.4 建议调整后的优先级

#### P0：稳定性

- 引入 SQLite 持久化 Store，替换纯内存 `MemoryStore`。
- 持久化 user、session、agent、task、message、event、approval、artifact。
- 增加 OpenClaw agent reconciliation：重启后按 `gatewayAgentId`、`workspace`、`userId` 找回映射。
- 空响应或 OpenClaw 超时不能标记为正常完成，应显示失败或可重试。

#### P1：可用性

- 主界面展示 OpenClaw / Agent / Profile sync 状态。
- Agent 设置页支持完整 profile 修改与重新同步。
- Artifact 不一定回到聊天流，但需要在文件空间或任务详情中有入口。
- README 增加本地启动、OpenClaw 连接、常见错误排查。

#### P2：演示与团队协作

- 是否保留 mock fallback 需要产品决策：演示稳定优先则保留，真实 OpenClaw 优先则降级为开发模式。
- 管理员页、技能市场、复杂插件管理可以后置。
- Timeline 可以作为任务详情的辅助页，不建议挤占聊天主界面。

### 13.5 偏差程度判断

整体偏差约为中高：

- 架构目标仍一致：本地 Web + API + OpenClaw Gateway + 插件回写。
- 主流程重点发生变化：从“mock/OpenClaw 都能演示”转为“真实 OpenClaw 连接和用户 agent 隔离优先”。
- 最大缺口仍是持久化：这直接影响系统稳定性和长期可用性。
- 前端展示策略发生变化：计划中的 Timeline/Artifact 内联展示不再符合当前 UI contract，需要重新规划到任务详情或文件空间。

因此，这份计划不应该废弃，但需要把 SQLite 持久化和 OpenClaw 状态恢复提到最前面，把 Timeline/Artifact 改为二阶段体验优化。

## 16. OpenClaw Session 历史展示（2026-06-03）

本次实现了将 OpenClaw 的历史 session 在玄知前端侧边栏中展示，并支持点击查看历史消息。

### 16.1 背景

`taskService.listTasksForUser()` 之前已经通过 RPC `sessions.list` 将 OpenClaw session 投影为 Task 并合并到侧边栏列表中。但点击这些 session 投影的 task 时，消息列表为空——因为 `messageService.listMessages()` 只从 MemoryStore 读取，历史 session 的消息只存在于 OpenClaw 磁盘上的 JSONL 文件中。

### 16.2 实现方案

**核心思路：** 不改变现有的 session→task 投影流程，只在消息加载时增加 JSONL 文件 fallback。

```
点击 session task
  → messageService.listMessages()
  → MemoryStore 有数据 (active session) → 直接返回
  → MemoryStore 无数据 (历史 session) → sessionService 读取 JSONL 文件 → 返回消息
```

**新增文件：**

| 文件 | 说明 |
|------|------|
| `apps/api/src/services/sessionService.ts` | 读取 OpenClaw 磁盘上的 session 索引和 JSONL 对话记录 |
| `apps/api/src/routes/sessionRoutes.ts` | `GET /api/sessions`、`GET /api/sessions/:id`、`GET /api/sessions/:id/messages` |
| `apps/web/src/services/sessionApi.ts` | 前端 session API 客户端 |

**修改文件：**

| 文件 | 修改 |
|------|------|
| `packages/shared/src/protocol.ts` | 新增 `SessionInfo`、`SessionMessage` 类型 |
| `apps/api/src/services/messageService.ts` | `listMessages()` 增加 JSONL fallback，注入 `sessionService` |
| `apps/api/src/app/dependencies.ts` | 组装 `sessionService`，传入 `messageService` |
| `apps/api/src/app/registerRoutes.ts` | 注册 `sessionRoutes` |
| `apps/web/src/types/protocol.ts` | 重新导出 `SessionInfo`、`SessionMessage` |
| `apps/web/src/components/assistant/Sidebar.tsx` | session task 显示 🕐 图标 + "历史" 标记 |
| `apps/web/src/styles/assistant/sidebar.css` | `.conversation-badge` 样式 |

### 16.3 JSONL 路径解析

OpenClaw session 文件存储在 WSL 中：

```
~/.openclaw/agents/{gatewayAgentId}/sessions/
  sessions.json              # session 索引 (key → metadata)
  {sessionId}.jsonl          # 对话记录 (每行一个 JSON 事件)
```

后端运行在 Windows，通过 `//wsl.localhost/Ubuntu/` UNC 路径访问 WSL 文件。
`sessionService.toWindowsWslPath()` 自动检测平台 (win32) 并转换路径。
